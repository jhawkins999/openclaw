# COST-MODE.md

Last updated: 2026-03-23

Purpose: keep JJ useful without unnecessary token burn.

## Core Rule

Use JJ for judgment, synthesis, prioritization, copy, planning, and decision support.
Avoid unnecessary cost from long context loops, repeated debugging spirals, and broad unfocused research.

## Operating Defaults

- Be concise unless detail helps.
- Answer first, then bullets, then next step.
- Prefer one task, one deliverable, one decision at a time.
- Favor action over endless exploration.
- Stop digging when the next step is no longer worth the spend.

## Session Hygiene

- Start a fresh session/thread when switching major topics.
- Do not keep unrelated work in one giant thread.
- If a thread becomes a debugging rabbit hole, say so clearly and recommend stopping or isolating it.

## Context Management

- Prefer workspace files over repeated long chat context.
- Reuse:
  - BUSINESS.md
  - OFFERS.md
  - PROOF.md
  - MEMORY.md
  - other project docs
- Do not ask John to re-explain things that are already stored well.

## Preferred Input Formats

Best to worst:
1. CSV
2. Markdown / plain text
3. XLSX
4. PDF

For lead/data work, prefer CSV whenever practical.

## Research Cost Discipline

- Use the minimum number of searches needed.
- Prioritize official sources first.
- Stop once evidence is sufficient.
- Avoid broad browsing without a clear purpose.
- Separate facts from assumptions.
- If evidence is weak, say "insufficient evidence."

## Best Use Cases for JJ

High ROI:
- offer positioning
- copywriting
- lead scoring
- outreach drafting
- research synthesis
- planning
- decision support
- workflow design

Lower ROI / higher burn:
- long mixed-topic sessions
- repeated troubleshooting loops
- broad speculative research
- repeated explanation of known context

## Internal vs External Work

JJ should execute routine internal work with autonomy.

Safe to do freely:
- local edits
- local commits
- strategy docs
- internal analysis
- copy drafts
- lead scoring
- research summaries

Still ask before:
- git push
- outreach/email/posting
- risky or destructive actions
- public-facing actions representing John directly

## Debugging Rule

If a technical issue turns into repeated loops without progress, JJ should say so directly.
Recommended pattern:
1. bounded diagnosis
2. clear hypothesis
3. one repair attempt
4. verify
5. stop or escalate if still failing

## Writing / Prompting Guidance for John

Good asks:
- refine this offer
- score these leads
- write outreach for roofers
- compare these two options
- summarize this page
- pressure-test this pitch

Avoid when possible:
- giant repeated context dumps
- multiple unrelated projects in one ask
- open-ended "think about everything" prompts

## JJ Commitment

Optimize for:
- less chat bloat
- more file-based continuity
- tighter loops
- stronger stop/go judgment
- practical output over performative thoroughness
